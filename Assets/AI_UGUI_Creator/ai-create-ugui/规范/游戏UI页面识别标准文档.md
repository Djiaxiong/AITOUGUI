# 游戏 UI 页面识别标准文档

Game UI Page Scope Recognition Spec

版本：v1.0

用途：用于指导 AI 根据用户一句需求，识别并输出一套游戏 UI 所需的页面集合。

适用范围：通用游戏 UI 规划、静态原型规划、UI 页面清单生成、AI 辅助设计前置分析。

## 1. 文档目标

本规范用于解决以下问题：

当用户只提供一句简短需求，例如：

`我要一套 RTS 独立单机游戏的 UI`

AI 的第一步不是直接画页面，也不是直接产出组件，而是应当先完成：

- 识别游戏类型
- 识别产品形态
- 识别页面覆盖范围
- 输出一套标准页面清单
- 标明哪些页面是必选、可选、类型扩展页

在页面集合确定后，才能进入 UI 设计执行阶段。

因此，本规范定义一套标准流程，用于将一句需求转化为：

- 页面结构树
- 页面清单
- 页面优先级
- 页面来源依据（通用 / 类型扩展 / 平台扩展）

## 2. 核心原则

AI 在处理游戏 UI 需求时，必须遵循以下原则：

### 2.1 先识别“页面范围”，再做设计

任何游戏 UI 设计任务，都应先明确该项目包含哪些页面，而不是直接开始设计单页。

### 2.2 页面识别分为三层

所有页面都来自以下三层组合：

- 通用基础页面
  - 几乎所有游戏都会有
- 玩法类型页面
  - 根据 RTS、RPG、卡牌、射击等类型追加
- 产品模式页面
  - 根据单机 / 联机 / 独立 / 商业化 / 移动端 / PC 等条件追加或删减

### 2.3 输出结果必须结构化

AI 不应只回答“会有主菜单、设置、战斗这些页面”，而应输出：

- 页面树
- 页面清单
- 页面分类
- 页面优先级
- 可选项说明

### 2.4 “一套 UI”默认指页面系统而不是单页

当用户说“做一套 UI”时，默认理解为：

一组完整支撑该游戏核心体验的页面系统。

除非用户明确说明只做某一页或某一模块，否则 AI 应优先给出整套页面范围。

### 2.5 skill 落地规则

本 skill 中，这一步必须落成 `source/page_scope.md`。

- 如果用户明确只做单页或单模块：
  - `source/page_scope.md` 仍然要生成
  - 但 `scopeMode` 标记为 `single-page`
  - 页面清单只保留目标页及其必要依赖页
- 如果用户要求整套 UI、完整 UI、某类游戏 UI，或需求模糊：
  - 默认进入 `full-suite` 或 `core-flow` 识别
  - 必须输出页面结构树、页面清单、最小页面集、完整页面集

后续 `source/site_plan.md`、`source/site.json.pages[]`、`source/page_plans/` 必须以 `source/page_scope.md` 为准。

## 3. AI 页面识别流程

当 AI 接收到用户一句需求时，应按以下顺序进行判断：

### Step 1：提取需求关键词

从用户输入中提取以下维度：

1）游戏玩法类型

例如：

- RTS
- RPG
- ARPG
- SLG
- FPS
- TPS
- 卡牌
- 回合制
- MOBA
- Roguelike
- 生存建造
- 模拟经营
- 放置挂机
- 体育竞速

2）运行模式

例如：

- 单机
- 联机
- MMO
- PvP
- PvE
- 合作
- 沙盒

3）产品定位

例如：

- 独立游戏
- 商业项目
- 手游
- PC 游戏
- 主机游戏
- 页游
- 轻度
- 中重度

4）范围修饰词

例如：

- 完整一套
- 只要核心流程
- 先做主界面
- 先做战斗部分
- 不含商城
- 不做活动页

### Step 2：判断 UI 范围模式

按下列规则判断 `scopeMode`：

- `single-page`
  - 用户明确指定单页或单模块
  - 例如：`先做主菜单`、`只做战斗 HUD`、`先做结算页`
- `core-flow`
  - 用户要一套 UI，但明确说只做核心流程
- `full-suite`
  - 用户要求完整一套、整套 UI、某类型游戏 UI
  - 或没有明确限制且需求模糊时，默认按整套 UI 处理

### Step 3：套用基础页面集

任何游戏，先从基础页面集开始。

```text
基础必备页
├─ 启动页 / Logo页
├─ 登录页 / 进入页
├─ 加载页
├─ 主界面 / 主菜单 / 主城 / 大厅
├─ 设置页
├─ 通用弹窗系统
└─ 异常状态页（更新、断线、报错等）
```

说明：

- 如果是纯单机 PC 独立游戏，可弱化登录页，但仍保留“进入页/标题页”
- 如果是纯离线独立游戏，可以将“登录页”替换为“开始游戏页”
- 如果是移动商业游戏，则登录页、公告、选服常常是标准项

### Step 4：根据玩法类型追加页面集

在基础页面集上，根据游戏类型追加一组玩法页面。

例如：

- RTS 追加：战役选择、任务准备、战斗 HUD、单位选择、建造菜单、结算页
- RPG 追加：角色页、装备页、技能页、地图页、任务页
- 卡牌追加：卡组编辑、角色养成、抽卡、阵容编排
- 射击追加：模式选择、武器配置、匹配页、战绩页

### Step 5：根据产品模式增删页面

根据项目的产品模式，再对页面集合做修正。

单机模式常见修正：

```text
删除：登录、选服、公会、好友、充值、运营活动
保留：开始游戏、继续游戏、存档、设置、任务、图鉴、结算
```

联机模式常见修正：

```text
增加：匹配、房间、好友、聊天、排行榜、公会、赛季
```

独立游戏常见修正：

```text
保留核心玩法相关页面
删除复杂商业化和社交运营页
UI 页数控制更克制
更强调主菜单、存档、图鉴、日志、设置、战斗流程
```

商业化项目常见修正：

```text
增加：商城、礼包、活动、签到、抽卡、月卡、通行证、邮件、公告
```

### Step 6：输出页面结构树

AI 必须输出：

- 页面分类树
- 页面列表
- 页面来源说明
- 页面优先级

## 4. 页面分类标准

为了统一识别规则，所有页面归属以下标准分类：

### A. 基础流程类

用于进入游戏和切换流程。

包括：

- 启动页
- 登录页 / 开始页
- 选服页
- 角色创建页
- 加载页
- 新手引导页

### B. 主导航类

用于承载整个系统的主入口。

包括：

- 主菜单页
- 主城页
- 大厅页
- 世界地图页
- 主导航页

### C. 核心玩法类

用于进入一局、执行玩法、结束反馈。

包括：

- 关卡选择页
- 模式选择页
- 任务选择页
- 编队页
- 战斗 HUD
- 暂停页
- 胜利结算页
- 失败结算页

### D. 角色与成长类

用于角色管理和养成。

包括：

- 角色列表
- 角色详情
- 技能页
- 装备页
- 升级页
- 天赋树
- 皮肤 / 外观页

### E. 资源物品类

用于道具、材料、装备管理。

包括：

- 背包
- 仓库
- 物品详情
- 合成页
- 分解页

### F. 目标与收集类

用于任务驱动与收藏表达。

包括：

- 任务页
- 成就页
- 图鉴页
- 剧情档案
- 收集进度页

### G. 经济商业化类

用于付费和资源消费。

包括：

- 商城
- 抽卡
- 礼包
- 通行证
- 充值页
- 活动兑换

### H. 社交多人类

用于多人和社交承载。

包括：

- 好友
- 聊天
- 邮件
- 公会
- 房间
- 匹配
- 排行榜

### I. 系统支持类

用于系统设置和异常处理。

包括：

- 设置页
- 帮助页
- 客服页
- 公告页
- 更新页
- 断线重连
- 协议 / 关于页

### J. 通用弹窗类

用于所有页面通用交互。

包括：

- 奖励弹窗
- 确认弹窗
- 物品详情弹窗
- 提示弹窗
- 错误弹窗
- 引导蒙层

## 7. 产品模式修正规则

- 单机
- 离线
- 本地
- 独立单机

删除或弱化页面：

```text
删除/弱化
├─ 登录页（改成开始页）
├─ 选服页
├─ 好友页
├─ 公会页
├─ 聊天页

```

增加或强化页面：

```text
增加/强化
├─ 开始游戏页
├─ 存档/读档页
├─ 难度选择页
├─ 战役/章节选择
├─ 图鉴/日志
├─ 暂停页
├─ 本地设置
```

## 8. 页面优先级定义

AI 输出页面时，必须给每个页面标记优先级。

### P0：绝对核心页

没有这些页面，无法形成完整体验。

例如：

- 开始页 / 登录页
- 主界面
- 核心玩法页
- 加载页
- 设置页
- 结算页

### P1：强推荐页

大多数同类游戏都会有，能明显提升完整度。

例如：

- 图鉴
- 任务
- 背包
- 存档
- 角色页
- 地图页

### P2：可选扩展页

取决于项目规模与预算。

例如：

- 成就
- 剧情档案
- 重播
- 排行榜
- 活动页
- 科技树页

### P3：条件页

只有在特定产品模式下才需要。

例如：

- 充值
- 公会
- 抽卡
- 月卡
- 赛季
- 聊天

## 9. AI 输出格式标准

AI 根据一句需求识别页面后，必须按以下格式输出，并写入 `source/page_scope.md`。

### 9.1 标准输出格式

```text
一、需求识别
游戏类型：
运行模式：
产品定位：
UI 范围判断：
scopeMode：

二、页面结构树
按分类列出页面树

三、页面清单
页面名 | 分类 | 优先级 | 来源

来源取值：
- 基础页
- 类型扩展
- 单机修正
- 联机修正
- 独立游戏修正
- 商业化修正
- 条件可选

四、推荐最小页面集
输出最小可落地版本

五、推荐完整页面集
输出完整版本

六、执行结论
- 本轮实际进入设计的页面范围
- 哪些页本轮先做
- 哪些页暂缓
```

## 10. 标准示例

示例输入：

`我要一套 RTS 独立单机游戏的 UI`

示例输出：

### 一、需求识别

- 游戏类型：RTS
- 运行模式：单机
- 产品定位：独立游戏
- UI 范围判断：应以单机 RTS 核心玩法流程为主，保留主菜单、战役、战斗 HUD、结算、图鉴、存档、设置等页面，去除联机社交与商业化页面
- scopeMode：full-suite

### 二、页面结构树

```text
RTS独立单机游戏UI
├─ A. 基础流程
│  ├─ 启动页
│  ├─ 开始游戏页
│  ├─ 加载页
│  └─ 存档/读档页
```

## 11. AI 执行规则总结

AI 在接收到“我要一套 XX 游戏 UI”时，必须按以下规则执行：

```text
1. 先识别游戏类型
2. 再识别运行模式（单机/联机）
3. 再识别产品定位（独立/商业化/手游/PC）
4. 再判断是单页、核心流程，还是整套 UI
5. 先套基础页面集
6. 再套类型扩展页面集
7. 再做产品模式增删修正
8. 输出结构树 + 页面清单 + 优先级
9. 确认页面范围后，才进入原型/视觉/UI设计阶段
10. 后续 site plan / site.json / page plans 必须引用这一结果
```

## 12. 可直接作为系统提示词的简版规则

如果你要喂给 AI 当系统规则，可以用这个简版：

```text
当用户要求“做一套游戏UI”时，不要直接开始设计页面内容。先判断该游戏UI应包含哪些页面。

识别流程：
1. 提取游戏类型（RTS/RPG/FPS/卡牌/SLG等）
2. 提取运行模式（单机/联机/PvP/PvE）
3. 提取产品定位（独立/商业化/手游/PC）
4. 判断范围模式：single-page / core-flow / full-suite
5. 先套用基础页面集：启动页、进入页、加载页、主界面、设置页、通用弹窗、异常页
6. 再根据玩法类型补充页面
7. 再根据单机/联机/独立/商业化做增删
8. 输出：
   - 页面结构树
   - 页面清单表
   - 每页优先级（P0/P1/P2/P3）
   - 页面来源（基础页/类型扩展/模式修正）
   - 最小页面集 / 完整页面集
9. 页面范围确认后，再进行后续UI设计
```
