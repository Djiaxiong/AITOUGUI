# AICreateUGUI Skill 使用指南

## 0. 先判断是不是“可执行环境”

这套 skill 不是单纯的提示词包，而是一套依赖本地工程和 Python 工具链的工作流。

先区分两件事：

1. 工作流通用
   - `Codex / Claude / GPT / GLM` 这类模型，只要能读文档、写文件、跑命令，都可以按这套流程工作

2. skill 装载方式不通用
   - `SKILL.md` 是 `Codex` 风格的本地 skill 入口
   - 其他代理如果没有同类机制，需要手动把 `SKILL.md + 规范/ + 使用指南` 注入为项目规则

### 最低环境要求

- 当前工程目录可读写
- 本地有可用的 `Python 3`
- 可以运行：
  - `generate_site_contract.py`
  - `validate_and_prepare_repair.py`
  - `compile_site_bundle.py`
  - `audit_compiled_theme_health.py`
- 如需最终落地 Unity，工程里还必须存在对应的 `AIToUGUI` 导入端

### 平台入口建议

- 直接以 `SKILL.md + 使用指南.md + 规范/` 作为通用入口
- 不再为某个单独模型维护独立工作流说明；模型差异只体现在执行能力，不体现在规则版本

### 不适合直接使用的场景

- 只有网页聊天框，没有本地文件系统权限
- 不能运行本地 Python 命令
- 不能持续修改目标站点包的 `source/` 并回跑验证

## 1. 这套 skill 到底做什么

这套 skill 不直接生成 Unity 代码。

它做的是：

1. 生成 HTML/CSS 站点包
2. 把站点包收敛到 Unity 可消费子集
3. 跑本地工具校验和修复
4. 编译出 `compiled_site_bundle.json`

然后再由 Unity 导入端导入。

这套 skill 现在明确采用规则驱动，不再建议按固定模板、固定示例或固定案例壳来跑。

同时要明确一点：

- 强模型可以更稳定地一次写对复杂布局
- 弱模型更容易写出“浏览器里勉强成立、Unity 里会膨胀”的半闭合 flex 结构

这不意味着问题只能归因于模型。正式链路必须用 planning、validator 和编译器约束把这类风险尽量前移消化。

发布前当前建议也要明确：

- Unity 侧已经被视为稳定落地层，而不是持续兜底修设计的主战场
- Skill 侧才是当前主要补强层
- 正式目标模型是中上能力模型，低能力模型不再作为发布适配范围

如果模型连 planning artifact 都维护不稳，不要继续靠补 prompt 硬顶，优先：

1. 改成 `single-page`
2. 或改成 `core-flow`
3. 或直接换更强模型

## 2. 正确使用顺序

在正式使用前，先判断这次任务属于哪一种模式：

1. `Single Page Text Mode`
   - 用户用文字描述一个页面
   - 对应 `scopeMode = single-page`，`inputMode = text-brief`
2. `Single Page Image Mode`
   - 用户给一张 UI 图要求复刻布局
   - 对应 `scopeMode = single-page`，`inputMode = image-reference`
   - 需要模型具备多模态能力
3. `Full Suite Mode`
   - 用户要一整套 UI 或多页完整流程
   - 对应 `scopeMode = full-suite`

`core-flow` 不是单独的主要产品模式，更适合作为整套 UI 难度过高时的收缩交付版本。

### 步骤 1

把这些文件给 AI：

- `SKILL.md`
- `规范/AIToUGUI_GenerationLauncher.md`
- `规范/UI风格参考.md`
- `规范/游戏UI页面识别标准文档.md`
- `规范/AIToUGUI_形态语言与风格分叉规范.md`
- `规范/AIToUGUI_SitePackageSpec.md`
- `规范/AIToUGUI_能力池与当前上限.md`
- `规范/AIToUGUI_Contract与自检规范.md`
- `规范/AIToUGUI_CSS与HTML实现细则.md`

### 步骤 2

让 AI 先做风格理解和页面规划，再做 Draft HTML，然后再做 Normalize。

如果用户说的是“做一套 UI”或只给了某类型游戏的模糊需求，先做页面范围识别，再做页面规划。
如果用户明确只做某一页，页面范围识别可以收成单页模式，但 `page_scope.md` 仍然要保留。

不要一上来就硬要求它直接写最终 bundle json。
也不要把“强化约束”留给最终用户补 prompt，应该由 skill 自己预先收口 authoring 规则。
更不要从旧页面、旧案例、旧模板里直接改配色开工。

推荐至少产出：

- `source/draft_visual_intent.md`
- `source/page_scope.md`
- `source/site_plan.md`
- `source/page_plans/<pageId>.md`

如果是 `full-suite`，还要强制要求：

- `site_plan.md` 写清 `shared invariants`
- `site_plan.md` 写清 `page delta matrix`

这里的目标不是只保证“看起来同风格”，而是同时保证：

1. 多页确实属于同一套 UI
2. 每页又有明确功能差异和骨架差异

至少要避免两类失败：

- 所有页面都长得一样，只是模块坐标变了
- 为了做出差异，反而每页像不同产品线拼在一起

### 步骤 3

生成站点包后，本地执行：

```powershell
python Assets/AI_UGUI_Creator/ai-create-ugui/PythonTool/generate_site_contract.py <SiteRoot> --write-self-check
python Assets/AI_UGUI_Creator/ai-create-ugui/PythonTool/validate_and_prepare_repair.py <SiteRoot>
python Assets/AI_UGUI_Creator/ai-create-ugui/PythonTool/compile_site_bundle.py <SiteRoot>
python Assets/AI_UGUI_Creator/ai-create-ugui/PythonTool/audit_compiled_theme_health.py <SiteRoot>
```

### 步骤 4

在 Unity 中导入根目录下的：

- `compiled_site_bundle.json`

## 3. 通过标准

满足这几个条件，才算合格：

1. `reports/validation_report.json.status == pass`
2. `reports/compile_report.json.status == pass`
3. 根目录存在 `compiled_site_bundle.json`
4. `preview.html` 能正常打开所有页面
5. `reports/theme_health_report.json` 没有明显 fallback 警告
6. Unity 导入后页面结构与视觉没有明显跑偏

对于 `full-suite`，发布前还建议额外人工确认两件事：

1. 是否存在“所有页面共用同一骨架，只做平移”的趋同问题
2. 是否存在“页间差异只靠配色和文案，页面任务感不成立”的问题

## 4. 当前和 Unity 落地端对齐的能力

已经稳定对齐：

- 多页站点包
- `site.json.designWidth / designHeight` 驱动的固定设计空间
- `:root` token、`.class`、`[data-ui-role]`
- 纯色、单层线性渐变
- 统一圆角、非均匀圆角、capsule
- 程序化边框、稳定阴影、glow
- `data-ui-shape`
- `data-ui-frame`
- `cut-corner / plate / banner`
- 显式 SVG 资产引用（icon / frame / ornament / rotated background）
- 基础节点语义、页面导入、Prefab 导出

部分对齐：

- `motion`
  - Unity 落地端已经有基础运行时动效
  - 当前 skill 侧不建议把最终视觉成立建立在动画 authoring 上

### SVG 正式路径

SVG 现在是正式支持路径，适合承接这几类视觉：

- `icon`、印章、圆环、徽记、法阵、复杂符号
- 旋转背景、非对称 ornament、复杂装饰底纹
- 外框、边框、按钮底板、复杂 `plate`
- 需要在 Unity 导入后转 Sprite / Nine Slice 的复杂图形

但它不是强制项。页面规划里如果没有 icon 或复杂视觉诉求，就继续走程序化 panel / button / text，不要为了补功能而硬塞 SVG。

不要把 SVG 用在这些地方：

- 普通纯色 panel / button / roundrect
- 需要运行时改单词或数值的文本排版
- 整页截图或整卡截图
- 依赖 `filter` / `mask` / `mix-blend-mode` 的浏览器特效

正确放置方式：

- SVG 文件放在 `source/assets/`
- 页面通过显式宿主节点引用 SVG，并补齐 `data-ui-asset-id/type/usage/import`
- 需要切片时补 `data-ui-asset-slice`
- 需要稳定尺寸时补 `data-ui-asset-width/height`
- 不希望被 Unity 染色时补 `data-ui-asset-tint="none"`
- HTML 预览直接消费 SVG；站点包 authoring 阶段不要产出 `sprites/`，最终栅格化由 Unity Editor 扩展完成

推荐宿主节点写法：

```html
<div
  data-ui-name="login-frame"
  data-ui-role="frame"
  data-ui-asset-id="login-frame-ornate"
  data-ui-asset-type="frame"
  data-ui-asset-usage="nineslice-frame"
  data-ui-asset-import="sliced"
  data-ui-asset-slice="48,48,48,48"
  data-ui-asset-width="600"
  data-ui-asset-height="480"
  data-ui-asset-tint="none"
  style="width:600px;height:480px;background-image:url('../assets/login-frame-ornate.svg');background-repeat:no-repeat;background-size:100% 100%;"
></div>
```

## 5. 当前不要做的事

不要依赖这些能力（完整口径以 `SKILL.md` 与 `检查清单.md` 为准）：

- 任意浏览器 `background-image`（仅显式本地 SVG 资产引用除外）
- `clip-path`
- `mask`
- `mix-blend-mode`
- `filter`
- `backdrop-filter`
- `grid`
- `calc()`
- `aspect-ratio`
- `flex-wrap`
- `position:fixed`
- `conic-gradient`
- `repeating-linear-gradient`
- `repeating-radial-gradient`
- 复杂 `@keyframes`
- 复杂渐变叠加

另外，不要依赖这些 authoring 习惯：

- 页内 `<style>`
- 越界装饰后靠裁切隐藏
- 复杂卡片直接写成 primitive `button`
- 自造 primitive button variant 名称
- 先写浏览器优先 HTML，再期待 normalize 自动救回来
- 自造一整套平行 role 体系，绕开 canonical role / text role
- 直接复用固定模板骨架、固定组件壳、固定案例排布
